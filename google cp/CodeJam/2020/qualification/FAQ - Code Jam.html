  • Overview

    keyboard_arrow_up

    • I've never done Code Jam; what is it?

      Code Jam is an annual tournament in which you write code to solve algorithmic problems that have multiple layers of difficulty.

    • Who is eligible to compete in Code Jam?

      See Section 1 (Eligibility) of the Coding Competitions Terms for complete eligibility criteria.

    • How does Code Jam work?

      1. Create a Coding Competitions profile if you don't have one already. Then, when registration for the contest opens up, sign up! You must have or create a profile before you're able to register for a contest.
      2. Once the Round starts, a link on the homepage will direct you to the dashboard for that Round.
      3. On the Round Overview page, you'll see the problems. They will be listed in approximate order of difficulty, so we recommend starting with the first one.
      4. Read the problem description:
        1. The statement describes the problem that you need to write code to solve.
        2. The Input and Limits sections describe the test sets, which are an ordered list of difficulty targets for your program to hit. These sections make guarantees about the sort of data your code will have to run on for each test set.
        3. The Output and Sample Cases sections clarify what your code has to do.
      5. Write a solution that you think can pass at least Test Set 1.
      6. If you'd like, you can test your solution on test cases of your choice.
      7. When you submit a solution, we will first run your code against the sample cases (for non-interactive problems). If it passes those, we will run the code against Test Set 1, and on the next test set if that passes, and so on.
      8. For Visible Verdict Test Sets, you will learn during the Round whether your code passed. For Hidden Verdict Test Sets, you will only find out at the end of the Round. You can submit as many times as you like, with minor penalties.
      9. Solve as many test sets for as many problems as you can before the Round ends. The more test sets you solve, the higher your score.

    • Will I receive a Code Jam certificate and when are they given out?

      As a Code Jam contestant, you will receive a certificate as long as you make at least one competitive attempt during the Qualification Round. You will receive your certificate after the last round of your participation. Certificates will be available on your Coding Competitions profile and will appear sometime after the round ends. Please note that both the Coding Competitions Terms and Code Jam Rules must be met in order to receive a certificate. Please note that both the Coding Competitions Terms and CJ Rules govern the issuance of any certificate.

    • Where can I submit feedback or other questions not answered here?

      If you have any questions not addressed in this FAQ or would like to submit feedback to the Code Jam team, please contact us at codejam@google.com and we will do our best to respond within two (2) business days. For any questions that require an answer while a Round is active, we will respond as soon as possible.

  • Registering

    • How do I register for Code Jam?

      Before participating in Code Jam, you must first have a g.co/codingcompetitions profile. After registration opens, and before the end of the Qualification Round, you can register for Code Jam at g.co/codejam. You can tell whether registration is open by checking the Code Jam schedule.

    • I am unable to sign in to register. What should I do?

      We suggest the following troubleshooting fixes:

      • Clearing your cache.
      • Enabling third party cookies. Our site only uses cookies from Google, but since we are using multiple Google-owned domains, these cookies may be considered third-party. They must be enabled for the site to work. Most browsers tell you which cookies are being blocked; consider whitelisting only Google cookies that you are comfortable with.
      • Using an incognito browser window.

      If you still cannot register, please send us an email at codejam@google.com.

    • How can I confirm that I am registered?

      You can confirm that you are registered by signing in and opening the Coding Competitions companion.

    • What if I registered for Code Jam in the past on the old platform?

      To compete in this year's Code Jam, you still need to create a Coding Competitions profile, even if you registered for Code Jam in past years. To claim your past nickname and certificates, register with the same Google account you used previously. You'll see your past information pre-filled in the profile creation form. If you'd like to use your previous nickname with a different account, please see below. Once you've created a Coding Competitions profile, don't forget you'll need to register for Code Jam. See more on registering above. If you encounter difficulties, please let us know at codejam@google.com.

    • What can I use for my nickname?

      Contestants should avoid creating nicknames that could be interpreted as offensive or obscene, or violate the intellectual property rights of others (in each case, as determined at Google’s sole discretion). You should not include and are strongly discouraged from including any personally identifying information (e.g., your legal name) in your nickname because your nickname will be publicly displayed on the scoreboard. For more information, see Section 2.2 (Nicknames) of the Coding Competitions Terms.

    • How can I change my nickname or otherwise update my profile?

      You may update your profile at g.co/codingcompetitions.

      You can do this as follows:

      2. Register using account B, and select nickname N.

      Be aware that another user may take nickname N between steps 1 and 2. Also be aware that we may not support this option indefinitely. Finally, as a reminder and in accordance with Section 2.1(E) of the Coding Competitions Terms, you may not use both accounts A and B to compete in the same contest otherwise you risk being disqualified.

    • How do I request my Coding Competitions profile and information be deleted?

      In accordance with Section 4.4(B) of the Coding Competitions Terms, when you are logged into a Contest website, you will be able to access, review, and update some of your personal data held by Google in connection with a Contest. If you would like to delete any of your personal data held by Google in connection with Google's Coding Competitions please email codingcompetitions@google.com.

  • Preparing

    keyboard_arrow_up

    • How can I practice?

      On the Archive page, you can access past Code Jam problems and submit solutions in practice mode. Both your practice and competitive submissions (if applicable) can be found at the top of each problem page.

      Practice mode attempts do not alter the original scoreboard for the round, and they will not be visible to others. Please be aware that practice mode attempts contribute toward the usual limit on multiple submissions within a short time window.

    • How can I view other contestants' code from completed rounds?

      You can click on a contestant's nickname in the corresponding scoreboard to see all of their in-contest attempts and code. We believe that there is something to learn from any attempt to solve a problem, not just correct or scoring ones, and multiple attempts can provide insight into a contestant's solving process. Incorrect submissions might contain useful examples of subtle mistakes to avoid, or ideas that might be useful in solving other problems.

    • How can I get better at algorithmic problem-solving?

      It is often possible to do well in our contests with only an open mind, familiarity with at least one programming language, and good problem-solving skills... but it is still useful (and sometimes essential) to have a well-stocked algorithmic toolbox! Make sure you get some practice with common techniques like dynamic programming, breadth-first search, and divide and conquer, and be ready to use basic data structures like priority queues. Be aware of common problems like integer overflow and stack overflow that can cause an otherwise correct solution to fail.

      For more challenging problems, it is helpful to study algorithms for more advanced techniques like bipartite matching and maximum flow… and there are many even more advanced and beautiful algorithms out there!

      You may find it helpful to read a textbook like CLRS, but we think the best way to pick up algorithmic programming contest experience is to participate in such contests, and participate often, solving problems from previous rounds. That firsthand experience will help you decide on what specific topics to read more about or take a class.

    • What should I know about older problems?

      For Code Jam problems prior to 2017, you will see references to Small and Large datasets. These loosely correspond to Visible Verdict and Hidden Verdict in the new system; see "What are Visible Verdict Test Sets and Hidden Verdict Test Sets?", below.

      On the previous Code Jam platform, contestants downloaded input files, ran their code locally on those files to produce output, and submitted those output files along with their code. There was a flat time limit of 4 minutes for Small datasets and 8 minutes for Large datasets. We will continue to port these problems to the current platform; be aware that the time limits on the current platform may be significantly tighter or looser than originally intended, and it may not be possible to solve certain problems in certain languages. When in doubt, we recommend C++ in these cases because of its speed.

      Moreover, the local testing tools from 2018 worked somewhat differently from the ones we currently use. We recommend looking at tools from interactive problems from 2019.

  • Competing

    keyboard_arrow_up

    • What OS and browser should I use?

      You are free to use whichever operating system you wish for solving the problems. The website should work with the latest versions of most modern browsers; please let us know if you encounter any issues.

    • What resources can I use?

      You're welcome to use any documents written before the contest started, including web pages, or code that you wrote or have a license to use. Other than during the Qualification Round, discussing the problems with other contestants, or anyone else, during the Contest is not allowed. Likewise, you aren't permitted to write about the problems until after a Round is over.

    • What is special about the Qualification Round?

      The Qualification Round runs for twenty-seven (27) hours. You may use as much or as little of that time as you would like.

      Unlike every other Code Jam Round, there is no limit on the number of potential advancers from the Qualification Round. You will advance to Round 1 if you earn at least 30 points, regardless of your penalty time. You may wish to review Visible Verdict Test Sets versus Hidden Verdict Test Sets and optimistic scoring; it doesn't hurt to score some extra points just in case!

      You will not be disqualified for working with other contestants during the Qualification Round, only. Even so, we ask that you do not post solutions publicly or otherwise spoil the problems for other contestants. Collaboration during any Rounds other than the Qualification Round is strictly forbidden; see Section 7.1(E) of the Coding Competitions Terms for more information.

    • A Round is starting soon. Where do I go to compete?

      Visit the schedule page to see the official start times and access the Rounds to compete. Those who are eligible to compete in the Round will see a link to the Round overview page once the Round has started.

    • There is a Round happening right now, but I can only watch. Why?

      There are a few possible reasons:

      • You are not logged in with the same Google Account that you used to register for Code Jam. If signed in, click the profile icon at the top right to see with which account you are currently logged in and visit the profile page to check your registration status.
      • You did not meet the criteria (specified in the CJ Rules) to advance to the Round currently taking place.
      • If this is Round 1B or Round 1C (a subround), perhaps you have already advanced to Round 2 in an earlier Round 1.

      If none of these apply to you, please contact us immediately at codejam@google.com and we will investigate.

    • Do I have to solve the problems in any particular order?

      You can answer the problems in any order you like. Looking at a problem does not start any timers, so feel free to read all of them at the start and then solve them in the order of your choice. There are different point values for each test set in each problem, and we generally give lower point values to the test sets that we think are the easiest.

    • How will you run my code on the different test sets?

      When you make an attempt, we will first run your code against the sample cases (for non-interactive problems). If your code does not pass these samples, your submission will not be tested on any of the test sets, and will not count as a penalty attempt.

      Then, we will run your code against Test Set 1. If it passes every test in that test set (within time and memory limits, etc.), we will run it on Test Set 2 (if there is one), and so on, in order. If your code fails a test set, we will not run it on test sets later in the order, even if it is possible that your code could have passed one or more of those test sets.

    • How does Code Jam handle real numbers?

      Sometimes a decimal number will appear as part of an input file or a correct output file. This section describes how they work.

      • In output files:
        • The problem will specify if real numbers are treated specially, and will likely link to this FAQ answer. For example, in a problem where the correct output is Case #1: 500 2.3, the problem may indicate that the second value doesn't have to be the exact string 2.3, but could be any number within 10-6; but it might not indicate that for the 500, in which case that would have to be exact. In that case, Case #1: 500 2.30000001 would be accepted, but Case #1: 500.0 2.3 would not.
        • Specially-treated real numbers must be expressed in decimal form, with the decimal point expressed as a . character, not a , character.
        • Two specially-treated real numbers are considered equal if their absolute or relative difference is smaller than some constant amount, typically 10-6. More details follow.
        • Our judge program will accept real numbers using an "e" or "E" character to indicate exponential notation, like 5e6 or 1.0e9 (but not like 1,0e9).
      • Unless the problem specifies otherwise, in input files:
        • Real numbers will be expressed in decimal form, with the decimal expressed as a . character, not a , character.
        • We will not use a "thousands" separator for integers or real numbers. One thousand will be expressed as 1000, 1000.000 or something similar; but not as 1,000.
        • We will not use exponential notation.
      • The "absolute or relative difference" check is implemented by code equivalent to the following Python code: 
              
          def IsApproximatelyEqual(x, y, epsilon):
            """Returns True if y is within relative or absolute 'epsilon' of x.

            By default, 'epsilon' is 1e-6.
            """
            # Check absolute precision.
            if -epsilon <= x - y <= epsilon:
              return True

            # Is x or y too close to zero?
            if -epsilon <= x <= epsilon or -epsilon <= y <= epsilon:
              return False

            # Check relative precision.
            return (-epsilon <= (x - y) / x <= epsilon
                or -epsilon <= (x - y) / y <= epsilon)
      • Note that although real numbers in output files are sometimes considered equivalent to other real numbers, real numbers in input files are considered exact. For example, if you are asked to output whether the input is less than 2.0, and the number provided in the input file is 1.9999999, then "true" is the only acceptable result.
      • Some examples:
        • Unless the problem specifies otherwise, the following will not express real numbers in input files: 2,000, 1,5, 6.022e23.
        • In an output file, if the correct output is 1000.0 and an absolute or relative difference of 10-6 is permitted, then the following will be considered correct: 1000.001, 1000, 999.999, 1000., 1e3, 10.00001e2, 9.99999E2 (all are within a relative distance of 10-6). The following will be considered incorrect: 1000.002, 1,000, 1000,0, 999.998.
        • In an output file, if the correct output is 0.001 and an absolute or relative difference of 10-6 is permitted, then 0.001001 and 0.000999 will be considered correct (both are within an absolute difference of 10-6).
        • In an output file, if the correct output is 1000 and there is nothing to indicate that token will be judged specially, then only the exact string 1000 will be considered correct.

    • How does Code Jam handle whitespace in input files?

      Unless otherwise stated in the problem's Input section:

      • When an input file has multiple values on the same line, every two adjacent values are separated by a single space character (ASCII 32 / 0x20).
      • Each line ends with a single newline character (\n).
      • There is no other whitespace.

    • How does Code Jam handle whitespace in output files?

      Any amount of non-line-break whitespace (defined as spaces and tabs) is treated as a single space. We also ignore leading and trailing whitespace on each line.

      Make sure that you have a space between any distinct components of the output as defined in the Output section, and as demonstrated in the Sample Output. Also, do not break up a component by adding a space.

      For example, if the Output section asks you to output Case #1: followed by two integers, and the Sample Output shows Case #1: 2 3, none of the following would be valid:

      • Case# 1: 2 3
      • Case # 1: 2 3
      • Case #1 : 2 3
      • Case#1: 2 3
      • Case #1:2 3
      • Case #1: 23

    • What if a test case has multiple correct solutions?

      If a test case has multiple correct solutions, you may output any one of them. Some statements may tell you to break ties in a specific way or give the solution in a specific order, in which case you have to abide by those rules. If after applying every rule about choosing a solution and formatting it in the statement there are still multiple options, then all of them will be considered correct.

    • What are Visible Verdict Test Sets and Hidden Verdict Test Sets?

      Each problem has a sample test set (except for interactive problems), followed by one or more Visible Verdict Test Sets, followed by zero or more Hidden Verdict Test Sets. These Terms refer to whether you will be able to see your results on the scoreboard during the contest. They do not refer to whether you will be able to see the input data itself -- see "Will I be able to see the contents of the test sets?", below.

      During the round, whenever you make an attempt, you will see how many Visible Verdict Test Sets it passed, but you will not see results for any Hidden Verdict Test Sets. After the round is over, for the attempt that is used for your score (see Section 4.3 of the Code Jam Rules), you will see that the Hidden Verdict Test Sets have either been skipped (because you did not pass a Visible Verdict Test Set), or have results that will not be revealed during the Round.

    • Why hide some results?

      This builds suspense and incentivizes contestants to continue solving problems throughout the entire Round. Even if it looks like another contestant has solved everything, they may have failed one or more Hidden Verdict Test Sets!

      After a Round, you can only see Hidden Verdict Test Set results for at most one submission (see Section 4.3 of the Code Jam Rules); since the Hidden Verdict Test Set parts of the other attempts cannot factor into your score, there is no need to evaluate them.

    • Will I be able to see the contents of the test sets?

      No. For each test set, the Limits section of a problem statement will tell you how many tests there can be and will make other guarantees about the test parameters, but you will not be able to see the content of the test sets itself, even after the Round.

    • What phases of judging do the time and memory limits apply to?

      The time limit applies to the sum of the compilation time (if your code is written in a compiled language) and the running time. We calculate the time taken by your submission as user time + system time. In a program which uses multiple processes and/or threads, the sum of the times taken for each process and/or thread will be used. This usually means a parallel computation will be slower than the same computation performed sequentially because of the overhead required. Additionally, the time limit applies to an entire test set, not each individual test case.

      The memory limit refers to the maximum amount of memory that can be in use at any given time during the entire process of compiling (if applicable) and running your code.

    • Are the questions available in other languages? Can I use a translator?

      Because programming problems often require very precise language, the risk of mistranslating a problem is too great, and the consequences are very severe: a single wrong word can give a very real disadvantage or advantage to contestants reading it. Because of this, we've decided not to make the problems available in more than one language. You may use the mechanical translator of your choice. Talking about the problems with other people, even translators, is not allowed.

    • How will I know if I've advanced to the next Round or if I'm eligible for a t-shirt?

      See Section 3.2 (Notification of Advancement) in the CJ Rules for details on how to know if you've advanced to the next Round. If you've advanced to Round 3 and are eligible to receive a Code Jam t-shirt, you will be notified by email, but note that you may only receive a t-shirt if you opt-in to receive prizes and provide in your Coding Competitions profile your name, phone number, a valid mailing address, and any other information Google may need to award or send you a prize.

    • How can I ask for help or report an issue during a Round?

      You can submit your question via the “Ask a question” button on the problem overview page. Keep in mind that in order to keep things fair for all contestants, we cannot provide hints or clarify the statements. We may share your question (anonymously, with the wording possibly edited for clarity) and our answer with all contestants.

  • Coding

    keyboard_arrow_up

    • How should my program read input and write output?

      Your program must read input from standard input (stdin) and write output to standard output (stdout).

      Consider the following very simple problem. The input consists of one line with a number of test cases T, and then T more lines, each of which contains two positive integers N and M. The desired output form is Case #x: y z, where x, y, and z are all integers; x is the case number (starting from 1), y is the sum of N and M, and z is the product of N and M.

      Here's some code to solve this problem, reading from stdin and writing to stdout, in various languages. For our examples, we use some of the most commonly used languages, but you are certainly not limited to using just these!

      C++ 

        
    #include <iostream> // includes cin to read from stdin and cout to write to stdout
    using namespace std; // since cin and cout are both in namespace std, this saves some text
    int main() {
      int t, n, m;
      cin >> t; // read t. cin knows that t is an int, so it reads it as such.
      for (int i = 1; i <= t; ++i) {
        cin >> n >> m; // read n and then m.
        cout << "Case #" << i << ": " << (n + m) << " " << (n * m) << endl;
        // cout knows that n + m and n * m are ints, and prints them accordingly.
        // It also knows "Case #", ": ", and " " are strings and that endl ends the line.
      }
      return 0;
    }

      Java 
        
    import java.util.*;
    import java.io.*;
    public class Solution {
      public static void main(String[] args) {
        Scanner in = new Scanner(new BufferedReader(new InputStreamReader(System.in)));
        int t = in.nextInt(); // Scanner has functions to read ints, longs, strings, chars, etc.
        for (int i = 1; i <= t; ++i) {
          int n = in.nextInt();
          int m = in.nextInt();
          System.out.println("Case #" + i + ": " + (n + m) + " " + (n * m));
        }
      }
    }

      Python 2 
        
    # raw_input() reads a string with a line of input, stripping the ' ' (newline) at the end.
    # This is all you need for most Code Jam problems.
    t = int(raw_input()) # read a line with a single integer
    for i in xrange(1, t + 1):
      n, m = [int(s) for s in raw_input().split(" ")] # read a list of integers, 2 in this case
      print "Case #{}: {} {}".format(i, n + m, n * m)
      # check out .format's specification for more formatting options

      Python 3 
        
    # input() reads a string with a line of input, stripping the ' ' (newline) at the end.
    # This is all you need for most Code Jam problems.
    t = int(input()) # read a line with a single integer
    for i in range(1, t + 1):
      n, m = [int(s) for s in input().split(" ")] # read a list of integers, 2 in this case
      print("Case #{}: {} {}".format(i, n + m, n * m))
      # check out .format's specification for more formatting options

      As you can see from these examples, most languages provide simple ways of reading from stdin and writing to stdout. You can always check the documentation of your favorite language to find the preferred way; almost all languages have one.

    • Are the time, memory, and other resource limits different for each language?

      The same limits apply to all languages. Although we aim to make most problems solvable in the stated time limits using most languages, it is not guaranteed that any problem can be solved in any language; relaxing the limits to accommodate the speed of a "slower" language might enable a suboptimal solution in a "faster" one. Just as in everyday software engineering, part of the contest is using the right tool for each job!

    • If I use Java, Kotlin, or Scala, how should I structure my code?

      You must name your main class Solution (with exactly that capitalization). It must contain a public main method. Furthermore, your code must not contain any package definitions. (Note: Package declarations are allowed in Kotlin.)

    • If I use PHP, how should I structure my code?

      You must enclose your source code with the standard opening (<?php) and closing (?>) tags.

    • May I use Unix utilities, e.g., invoking sed or cat via bash?

      Generally speaking, common Unix tools are available on our judging system, and you can even try to, e.g., use Bash to compile your code with different flags, if you are so inclined. However, you use such features at your own risk -- i.e., the tool you want may not be installed, or your code may not have permission to run it. We suggest you experiment in advance on practice problems, rather than during a contest while under time pressure.

    • What development environments can I use?

      You can use any text editor or development environment, but be advised that if you use an online integrated development environment (IDE), it is your responsibility to make sure nobody else can see your code during the Round. Be aware that for some popular online IDE websites, your code is publicly visible by default; in that case, you must change that setting so that your code for a Round is not publicly visible while that Round is still in progress. If we discover that a contestant has accessed and used another contestant's code, even without that contestant's knowledge, our policy is to disqualify both contestants.

    • Will the code I type or upload into the editor be saved?

      We currently save this code only in your browser's local storage, so it may not persist if your cache is cleared, for example, and it will not be present if you use a different computer. We strongly recommend periodically backing up your work to more stable storage on your own machine.

    • Does the editor compile or otherwise evaluate my code?

      The editor does not compile your code. It may point out some simple syntax mistakes, but this checking is not exhaustive and you should not rely on it. The editor also provides syntax highlighting for some languages.

    • During the round, can I use code written before the round?

      As long as you have a license to use it, yes. That means you can write your own helper code, or collect your own personal library of code, as long as the license terms of the code permit it. Some contestants will have competed on an ACM ICPC team, and many of those teams have their own code books; check with your team's coach whether it's permissible for you to use that code in a different context.

    • What is considered cheating?

      Collaborating with anyone else during any CJ Round (except for the Qualification Round) is strictly prohibited and will result in your disqualification. This includes discussing or sharing the problem statements or solutions with others during the Round. Participating with multiple accounts is also prohibited. If we believe that you have undermined the integrity of the CJ Contest, we reserve the right to disqualify you. Use your best judgment. If you have a question about whether something is allowed, or if you suspect another contestant of cheating, please contact us at codejam@google.com.

      See Section 7 (Disqualification) of the Coding Competitions Terms and Section 7 (Disqualification) of the CJ Contest Rules for more details.

  • Platform

    keyboard_arrow_down

    • What are the platform specifications?

      Our platform uses a 64-bit version of Debian 9.8 (stretch), and we install packages with:

      apt-get install -y --no-install-recommends <package>

      • Linux Kernel
        • Linux 4.9 for 64-bit PCs: 4.9.0-8-amd
        • package: linux-image-amd64
      Debian stretch has a good track record of being bug free, so we stick to the default packages as much as possible in order to mitigate risk. However, this causes the versions of some languages (e.g., Go) to be far from the newest.

    • What are the stack size limits?

      The stack size limit is 64 MB per language, defined by `ulimit -s 65536`.
      JVM-based languages are passed the `-Xss64m` flag during execution.

  • Testing

    keyboard_arrow_down

    • How can I test my code locally?

      Please check the Platform section for details on the operating system, interpreters and compilers our system uses to evaluate your code. We suggest you try to replicate that as much as possible. If you have other Unix based system (Linux, MacOS) it is likely that you can install the exact same interpreter/compiler and version we have. Most of them also have versions for Windows that are almost the same. We cannot take responsibility for differences, so be sure to check the documentation. After you have your code ready in a local file, you can run it using a local file for input. You can start by using the sample input provided in the statement, and also add your own tests to check for tricky cases. Remember that the sample input is not expected to cover every possible case your code should handle.

    • Can I test my code on the Coding Competitions platform?

      Yes -- click the “Show test input” toggle at the bottom of the code editor to test your code server-side on our platform. You will then be prompted to provide an input and will see a new “Run tests” button at the bottom of the page. For interactive problems, you will need to provide a custom judge in the form of code and language, instead of input. Click the “Run tests” button to run tests against your code with your specified input or custom judge. Be aware that the system will not check the validity of your test cases or judge against any limits or specifications. Remember that there is a 10 second time limit for each test run, regardless of the time limits stated in the problem, so you may need to limit the data size and the number of cases you add for each run.

    • Will I be penalized for my test submissions?

      You will not receive any penalty attempts for test submissions and they do not count towards your score during a round. Please be aware that running tests does contribute towards the usual limit on multiple submissions within a short time window.

    • What information will I receive about my test attempts?

      A new test submission entry will be added to the submission list at the top of the problem page. Once judgement has been completed, you will either see an error (see "What information will I receive about my attempts?", below), or a button to view the output for the test. To review the code or input used for a test submission, click on the eye icon next to it.

      "Completed" means that your code ran against your provided input and produced output. For test submissions, unlike regular submissions, you will never see verdicts like Wrong Answer or Correct, because correctness of the output is not evaluated. Be aware that if your output is too large, it may be truncated.

  • Interactive Problems

    keyboard_arrow_down

    • What is an interactive problem?

      In an interactive problem, our judging system and your code run at the same time. When you send output to your output stream, the judge reads that output and then responds by sending input to your input stream. Then your code reads that input, and determines what output to send next, and so on. Each test set proceeds in this way. Generally, the judge tells your code when a test set has been solved, or when you have failed the test set.

      For example, one classic interactive problem that you may have seen in real life is a number guessing game in which you have a certain number of tries to figure out a number that a friend has chosen. You make a guess, and your friend tells you whether the guess is too high, too low, or correct. If you are not correct, then, based on that information, you can make another guess, and get a response; this continues until you either guess the right number, run out of tries, or make an invalid guess like a non-number that causes your friend to get mad and stop playing.

    • How do interactive problems work?

      Just as in non-interactive problems, you will receive input from stdin and print output to stdout. Our system will do the job of directing your output stream to the judge's input stream, and pointing the judge's output stream to your input stream. However, every time you output data, you must flush your output buffer, as explained below.

      Anything your program sends through standard error is ignored, but it might consume some memory and be counted against your memory limit, so do not overflow it.

      If your program continues to wait for the judge after receiving an indication that your answer is incorrect (as described in the problem statement), your program will time out, resulting in a Time Limit Exceeded error. Notice that it is your responsibility to have your program exit in time to receive a Wrong Answer judgment instead of a Time Limit Exceeded error. As usual, if the total time or memory is exceeded, or your program gets a runtime error, you will receive the appropriate judgment.

      You should not send additional information to the judge after processing all test cases. In other words, if your program keeps printing to standard output after the last test case, you will get a Wrong Answer judgment.

    • What does it mean to flush a buffer?

      Many programming languages choose to "buffer" the output of your program. That is, when you send output, instead of being printed immediately to the output stream, it may be stored in a buffer. That entire buffer is then printed to the output stream at some later time (usually when your code terminates).

      Buffering is a great language feature in most situations, but it interferes with interactive problems! If you print output without flushing, and the output gets buffered, the judge will wait forever for output that never comes, and so your code will also wait forever for a judge response that never comes, causing a deadlock.

      You will need to manually flush the buffer every time that you print output, in order for the judging system to see it. Different languages have different syntax for output flushing. You can learn more by consulting the specification of the programming language of your choice.

    • Is there a sample interactive solution in my favorite language?

      The analysis for the Number Guessing problem from the Code Jam 2018 Practice Session includes sample solutions in several languages. Check it out! We hope that the overall pattern can easily be adapted to any language of your choice.

    • How can I test an interactive solution?

      You will need to be able to run Python locally.

      At the bottom of the statement for each interactive problem, you can find a link to download a judge. These judges are not guaranteed to behave exactly the same way as the ones that will evaluate your problem -- in particular, they may implement randomness in a different way. However, they should help you test your solution.

      To test your solution locally, use our interactive_runner script in conjunction with the judge. Instructions are included in the comments.

      To test your solution on our server, toggle "Show Test Code" on the bottom of the problem page. Add the judge to the bottom pane and select the appropriate language (generally Python 3). Also modify the judge to include the test cases of your choice. Then add your solution to the top pane, make sure to select a language for your code (it may be a different language than the judge's), and click "Run tests".

      When running remotely, the judge will be passed a 0 as a command line argument. If you are using one of our judges, this means the judge will run your code against whatever cases you have specified for the first test set.

      Be aware that the system will not check the validity of your test cases or judge against any limits or specifications.

      Remember that there is a 10 second time limit for each test run, regardless of the time limits stated in the problem, so you may need to limit the data size and the number of cases you add for each run.

      For practicing with Code Jam 2018 or 2019 problems, you can use the local testing tool included with each problem.

  • Submitting

    keyboard_arrow_down

    • What are the requirements for my source code?

      See Section 4.2 (Submitting Solutions for Code Jam Problems) of the CJ Rules.

    • The submit button is grayed out / not responsive.

      Please ensure that you have selected a language for your submission (and for the judge, if you are testing an interactive problem).

    • The judge is taking a long time to return a response.

      Check the time limits for the problem; if they are very large, it may take some time to fully evaluate an attempt. Also, it is possible that your attempt has been judged and your browser has not automatically refreshed, so if it has been a long time, try refreshing it manually. Otherwise, the delay may be due to high system load, and we recommend that you continue to work on the problem (or other problems) as you wait.

    • What information will I receive about my attempts?

      For each attempt, you will see symbols for any sample test sets or Visible Verdict Test Sets that the attempt passed, and/or for the first sample test set or Visible Verdict Test Set that the attempt failed. If the attempt failed a Visible Verdict Test Set, you will see an error code for the failed test. If there was a Compilation Error, you will also see information returned by the compiler.

        The possible error messages are as follows:
      • Compilation Error: Your code produced an error while being compiled. Formally, the compilation did not finish with exit code 0.
      • Compilation Time Limit Exceeded: Your code failed to compile within the stated time limit for the test set.
      • Compilation Memory Limit Exceeded: Your code used more memory than the stated memory limit for the test set while being compiled.
      • Runtime Error: Your code produced an error while running. Notice that some languages may detect that the process is about to exceed the memory limit before it occurs and emit a runtime error instead. Formally, you will receive a Runtime Error if your code does not finish with exit code 0.
      • Time Limit Exceeded: Your code failed to run to completion within the stated time limit for the test set. Please see the Competing section for more information on how time limits are handled.
      • Memory Limit Exceeded: Your code used more memory than the stated memory limit for the test set while running.
      • Wrong Answer: Your output for the test set was incorrect. See the Interactive Problems section for other situations that can cause this error.

      You will not see any results for Hidden Verdict Test Sets until the end of the Round, and even then, only for your last attempt for each problem. If your code fails the sample test set, you will see "Sample Failed:" followed by one of the above errors.

    • Why do I see a "test set skipped" result?

      If your attempt does not pass one of the test sets, it will not be run against any test sets later in the sequence, and you will see "test set skipped" for those sets.

      For example, suppose that a problem has a sample test set and two test sets, and you submit an attempt that passes the sample test set but gets a Runtime Error on the first test set. No verdict will be displayed for the sample test set, because verdicts are only reported for failed sample test sets. You will see "Runtime Error" for the first test set, followed by "Test set skipped" for the second test set.

    • If an attempt fails a test set, will I know how many of the tests in that test set it passed?

      No. We realize that it can be frustrating not to know how close you were to passing, but we do not wish to encourage "error-driven development".

    • If an attempt gets a Runtime Error, will I be able to see debug output, e.g., a stack trace? What about after the Round is over?

      We cannot provide this information because it could be "gamed" to try to gain unauthorized information about our input. Also, again, we do not wish to incentivize relying on error messages.

      After a Round is over, if you are having trouble debugging a solution, we suggest asking on our public Google group, where other contestants may be willing to help.

    • I accidentally submitted the wrong code, or indicated the wrong language. Is there a way to cancel or nullify that attempt?

      There is no way to cancel an attempt. However, if the Round is still running, you can make another attempt for that problem. It is to your advantage to do this if you are sure that your previous attempt was wrong; see the Scoring section for more details.

    • Once I make an attempt, how soon can I make another?

      You may not make more than 10 attempts in any 10-minute window. That is, if you have made at least 10 attempts, the tenth most recent of those must be at least 10 minutes old before you can make another attempt. This limit is system-wide, not specific to each problem.

    • What happens if I make an attempt close to the end of the Round?

      Any attempt received by our server before the end of the Round will be judged. Judging will continue as necessary even after the Round ends.

  • Scoring

    keyboard_arrow_down

    • How is my score for a particular problem determined?

      See Section 4.3 of the Code Jam Rules. As a summary:
      Your score for a problem consists of a point total and a penalty time. The penalty time is partly determined by the number of penalty attempts.
      For each problem, we choose one of the following as your scoring attempt:

      • Your last attempt that correctly solved all Visible Verdict Test Sets. In that case, we use the sum of the values of all passed test sets (Visible Verdict and Hidden Verdict) as your point total.
      • Your earliest attempt in which you earned the highest sum of values for correctly solved Visible Verdict Test Sets. In that case, we add up the scores for those passed Visible Verdict Test Sets.

      We choose the option that gives you the most points. If both give you the same number of points, we choose the second one, since it will give you an earlier penalty time, which is better. The idea here is that once you have solved a Visible Verdict Test Set, those points are yours to keep regardless of what happens on your future attempts. However, this is not true of Hidden Verdict Test Sets.

      In either case, your penalty time is the time at which you made your scoring attempt, and your number of penalty attempts is the number of attempts before your scoring attempt, regardless of how many tests sets they passed.
      However, we do not want to penalize you for any attempts following your scoring attempt, since they did not help you attain that score.

      For example, consider a problem with four test sets: two Visible Verdict Test Sets (I and II) followed by two Hidden Verdict Test Sets (III and IV). Suppose that your attempts, in order, are as follows:
      1. Passes I, but fails II.
      2. Passes I, II, and III, but fails IV.
      3. Passes I, II, III, and IV.
      4. Passes I and II, but fails III.

      In this case, we would use attempt 2 as your scoring attempt, and you would earn points for the first two test sets. Your penalty time would be the time of attempt 2, and you would have 1 penalty attempt to reflect the 1 attempt before that. Note that it does not matter that attempt 3 would have earned full points; the only way to get points for Hidden Verdict Test Sets is for your latest attempt to solve them (that also passed all Visible Verdict Test Sets).

      If you never pass any test sets for a problem (either because none of your attempts are successful, or because you make no attempts), your score for the problem is 0, your penalty time is 0, and your number of penalty attempts is 0.

    • What does "Sample Failed" mean?

      For non-interactive problems, if your code fails to pass the sample test cases provided in the statement, you will see a Sample Failed verdict, followed by one of the verdicts mentioned above (e.g., RE for Runtime Error).

      Attempts that do not pass sample cases are completely ignored for scoring. They are not counted as penalty attempts, and are not counted as your scoring submission. When the scoring rules refer to "latest attempt" or "earliest attempt", they mean the latest/earliest from the set of attempts that pass the sample cases. Notice that this new interpretation can only benefit you by increasing your score and/or lowering your penalty.

      Keep in mind that all attempts -- including tests, practice, and competitive attempts that do not pass sample cases -- count towards your limit on multiple submissions within a short time window.

    • Is there partial credit if my code passes only some of the tests within a test set?

      There is no partial credit within a test set, because we do not want to incentivize writing partially correct code. However, most problems have multiple test sets, so there are opportunities to earn some points for code that correctly solves easier test sets.

    • How is the winner of a Round determined?

      Your score for a Round consists of a point total and a penalty time.

      Your point total is the sum of your point total from each problem, as described above.

      Your penalty time is the latest among all penalty times from all problems that you solved, plus 4 minutes times the total number of penalty attempts across all problems.

      For example, suppose that a Round has 3 problems:

      • On problem 1, you scored 30 points, with a penalty time of 1:23:45, and 2 penalty attempts.
      • You attempted problem 2 but never solved a test set; you scored 0 points, with a penalty time of 0 and no penalty attempts.
      • On problem 3, you scored 15 points, with a penalty time of 0:12:34, and 1 penalty attempt.

      Your score would be 45 points, with a penalty time of 1:35:45.

      Ranks are determined based on the point total (with higher totals being better), with penalty time as a tiebreaker (with lower penalty times being better). Notice that it is always to your advantage to earn more points, regardless of when in the contest you earn them or how many penalty attempts you accrue in doing so.

      We use finer time precision than the scoreboard displays, so even if two contestants on the scoreboard appear to have the same penalty time in seconds, we have ranked them appropriately.

      See Section 4.3 (Judging and Scoring) of the of the CJ Rules for more information.

    • Who appears on the scoreboard?

      Anyone who has participated in a Round by making at least one attempt will appear on that Round's scoreboard, even if none of those attempts passed any test sets. Even if you are signed in and qualified for the contest, you will only see yourself on the scoreboard if you have made at least one attempt.

      Contestants with 0 points do not formally have a rank and can never advance to the next round.

    • Why is the scoreboard giving me points for a Hidden Test Set? / Why did my score decrease once the Round ended?

      During the Round, the scoreboard is "optimistic" — that is, it calculates your score (via the same scoring rules described above) assuming that all Hidden Verdict Test Sets that have not been skipped are correct. This may not be the case!

    • Why do some users lack flags on the scoreboard?

      Choosing a flag during registration is not mandatory. If you lack a flag and you want one, you can update your Coding Competitions profile.

    • After a Round is over, can I see the official input and correct output for a test set?

      Our current policy is not to provide that information. We want to encourage you to debug using your own samples rather than official test sets.

    • Why do attempts with a verdict of Compile Error count towards my penalty in interactive problems?

      Interactive problems do not have sample cases, so any rejected submission will penalized regardless of the verdict. Please ensure your code compiles on our platform by using the server-side testing feature before submitting a competitive attempt.
